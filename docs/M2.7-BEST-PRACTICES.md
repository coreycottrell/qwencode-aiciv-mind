## 1. Model Overview

**Architecture**: Mixture-of-Experts (MoE), 230B total parameters, ~10B active per token.
**Context window**: 200K tokens (input), 131K tokens (max output for M2.7).
**Key strength**: Agentic tool calling. Scored 76.8% on BFCL Multi-Turn (vs Claude Opus 4.6 at 63.3%).
**Cost**: $0.30/$1.20 per million tokens (input/output) — roughly 17-21x cheaper than Claude Opus 4.6.
**Inference**: Interleaved thinking model with `<think>` tags. Cannot disable thinking.

## 2. Tool Calling Format — The XML Schema

M2.7 uses a **proprietary XML format** for tool calls, NOT the standard OpenAI JSON format natively. When served through vLLM/SGLang with `--tool-call-parser minimax_m2`, it gets translated to OpenAI-compatible format.

### Native XML Output Format

```xml
<minimax:tool_call>
<invoke name="get_weather">
<parameter name="location">San Francisco, CA</parameter>
<parameter name="unit">celsius</parameter>
</invoke>
</minimax:tool_call>
```

### Multiple Simultaneous Tool Calls

```xml
<minimax:tool_call>
<invoke name="search_web">
<parameter name="query_list">["OpenAI latest release"]</parameter>
<parameter name="query_tag">["technology"]</parameter>
</invoke>
<invoke name="search_web">
<parameter name="query_list">["Gemini latest release"]</parameter>
<parameter name="query_tag">["technology"]</parameter>
</invoke>
</minimax:tool_call>
```

### Internal Prompt Template (how tools are injected)

The chat template wraps tool definitions like this:

```
# Tools
You may call one or more tools to assist with the user query.
Here are the tools available in JSONSchema format:

<tools>
<tool>{"name": "tool_name", "description": "...", "parameters": {...}}</tool>
</tools>

When making tool calls, use XML format to invoke tools and pass parameters:
```

### Tool Definition Format (OpenAI SDK)

```python
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "Get the current weather in a given location",
        "parameters": {
            "type": "object",
            "properties": {
                "location": {"type": "string", "description": "City and state, e.g., 'San Francisco, CA'"},
                "unit": {"type": "string", "enum": ["celsius", "fahrenheit"]}
            },
            "required": ["location", "unit"]
        }
    }
}]
```

### Tool Definition Format (Anthropic SDK)

```python
tools = [{
    "name": "get_weather",
    "description": "Get weather of a location",
    "input_schema": {
        "type": "object",
        "properties": {
            "location": {"type": "string", "description": "The city and state"}
        },
        "required": ["location"]
    }
}]
```

### Parsing Raw XML Output (when not using vLLM/SGLang)

MiniMax provides a reference parser. Key regex patterns:

```python
import re, json

def parse_tool_calls(model_output, tools=None):
    if "<minimax:tool_call>" not in model_output:
        return []

    tool_call_regex = re.compile(r"<minimax:tool_call>(.*?)</minimax:tool_call>", re.DOTALL)
    invoke_regex = re.compile(r"<invoke name=(.*?)</invoke>", re.DOTALL)
    parameter_regex = re.compile(r"<parameter name=(.*?)</parameter>", re.DOTALL)

    tool_calls = []
    for block in tool_call_regex.findall(model_output):
        for invoke in invoke_regex.findall(block):
            name_match = re.search(r'^([^>]+)', invoke)
            if not name_match:
                continue
            function_name = name_match.group(1).strip().strip('"').strip("'")

            params = {}
            for pmatch in parameter_regex.findall(invoke):
                pm = re.search(r'^([^>]+)>(.*)', pmatch, re.DOTALL)
                if pm:
                    pname = pm.group(1).strip().strip('"').strip("'")
                    pvalue = pm.group(2).strip()
                    params[pname] = pvalue  # type-convert based on tool schema

            tool_calls.append({"name": function_name, "arguments": params})

    return tool_calls
```

## 3. Interleaved Thinking — The Critical Mechanism

**This is the single most important thing to get right.**

M2.7 is an interleaved thinking model. It generates `<think>...</think>` blocks between tool calls. These blocks contain plans, hypotheses, constraints, and intermediate conclusions.

### The Rule: ALWAYS Preserve Think Tags in Conversation History

When passing conversation history back to the model:
- **DO NOT strip `<think>` tags** from assistant messages
- **DO NOT modify the `content` field** of assistant responses
- **Append the COMPLETE assistant response** to message history

### What Happens When You Drop Think Tags

| Benchmark | With History | Without | Degradation |
|-----------|-------------|---------|-------------|
| SWE-Bench Verified | 69.4% | 67.2% | -3.3% |
| Tau^2 | 87 | 64 | -35.9% |
| BrowseComp | 44.0% | 31.4% | -40.1% |
| GAIA | 75.7% | 67.9% | -11.5% |
| xBench | 72.0% | 66.0% | -9.1% |

BrowseComp drops by 40% when prior thinking is lost. This is catastrophic for agent workflows.

### Using reasoning_split (OpenAI SDK only)

```python
response = client.chat.completions.create(
    model="MiniMax-M2.7",
    messages=messages,
    tools=tools,
    tool_choice="auto",
    extra_body={"reasoning_split": True}  # Separates thinking into reasoning_details
)

# CRITICAL: Preserve reasoning_details when appending to history
msg = response.choices[0].message
messages.append({
    "role": "assistant",
    "content": msg.content,
    "reasoning_details": msg.reasoning_details,  # MUST preserve this
    "tool_calls": [tc.model_dump() for tc in (msg.tool_calls or [])]
})
```

Without `reasoning_split`, thinking appears as `<think>` tags embedded in the `content` field. Do not strip them.

### Why OpenAI-Compatible APIs Break This

The standard OpenAI Chat Completions API does not support passing `reasoning_details` back in subsequent requests. Many frameworks silently strip this field. This is the root cause of degraded M2.7 performance in most integrations.

**Anthropic-compatible API** handles this better — it natively supports interleaved thinking blocks.

## 4. Recommended Inference Parameters

### Official MiniMax Recommendations

```python
{
    "temperature": 1.0,
    "top_p": 0.95,
    "top_k": 40,       # M2.5/M2.7 (some docs say 20 for original M2)
    "max_tokens": 4096  # For typical tool-calling tasks; up to 131072 for M2.7
}
```

**Why temperature=1.0?** This is NOT a typo. M2.7 was trained with temperature 1.0 and its performance degrades at lower temperatures. The model uses its internal thinking to maintain coherence — lowering temperature restricts the diversity of reasoning paths.

### Self-Evolved Parameters (discovered by M2.7 itself)

During its self-evolution loop, M2.7 systematically searched for optimal sampling parameters and found improvements through tuning combinations of:
- temperature
- frequency_penalty
- presence_penalty

The exact values discovered were not published, but the principle is clear: the model benefits from controlled diversity in its sampling.

### Reasoning Budget

**There is currently NO official way to control reasoning effort in local deployments.**

- `reasoning.effort: "high"` works on OpenRouter but is not supported in vLLM/SGLang
- System prompts like "use LOW reasoning mode" are **ignored** — the model still fills think blocks
- Reducing `max_tokens` causes truncation of the final answer, not reduced thinking
- This is an unresolved issue: [GitHub #52](https://github.com/MiniMax-AI/MiniMax-M2/issues/52)

**Practical workaround**: Set `max_tokens` high enough (4096-8192 for tool calling, higher for code generation) to avoid truncation. Accept the thinking overhead as the cost of quality.

## 5. System Prompt Best Practices

### Be Explicit and Structured

M2.7 requires detailed, structured prompts. It cannot infer patterns from minimal instructions like Claude can.

**The STRUCTURE-CONSTRAINTS-NAMING-CONTEXT-DON'T Framework:**

```
STRUCTURE: [Define expected output sections/components]
CONSTRAINTS: [Specify tech stack, patterns, limits]
NAMING: [Exact file/component/variable names]
CONTEXT: [Reference existing patterns in the project]
DON'T: [Explicit exclusions to prevent creative drift]
```

### Tell the Model "Why"

When the model understands your purpose, it provides more accurate answers. Explain the reasoning behind constraints.

BAD: "Respond in under 60 characters."
GOOD: "Respond in under 60 characters because this feeds a text-to-speech system with a hard buffer limit."

### Show Examples

M2.7 generalizes well from examples. Provide a "template" example of desired output. Explicitly highlight mistakes to avoid.

### For Tool-Calling Agent Loops

Include in the system prompt:
1. Available tools and their purposes (the framework handles this, but reinforcing helps)
2. That the model is in a loop and can make multiple tool calls
3. When to stop calling tools and provide a final answer
4. Output format expectations between tool calls

### Keep Token Budget in Mind

"Keep the total input and output tokens within 200k tokens." Use context compression for single tasks. Restart fresh for multiple independent tasks.

### One Task Per Prompt

Keep prompts focused on one task at a time but include all needed context (key files, requirements). The long context window supports cross-file tasks, but scattershot prompts degrade quality.

## 6. Multi-Turn Tool Calling Patterns

### The Correct Multi-Turn Flow

```python
messages = [
    {"role": "system", "content": system_prompt},
    {"role": "user", "content": user_request}
]

while True:
    response = client.chat.completions.create(
        model="MiniMax-M2.7",
        messages=messages,
        tools=tools,
        tool_choice="auto"
    )

    assistant_msg = response.choices[0].message

    # CRITICAL: Append the COMPLETE assistant message (including think tags)
    messages.append(assistant_msg)  # Preserve everything

    if not assistant_msg.tool_calls:
        break  # Model decided no more tools needed

    # Execute each tool call
    for tool_call in assistant_msg.tool_calls:
        result = execute_tool(tool_call.function.name,
                              json.loads(tool_call.arguments))

        messages.append({
            "role": "tool",
            "tool_call_id": tool_call.id,
            "content": json.dumps(result)
        })

    # Loop continues — model sees tool results and decides next action
```

### Tool Result Format (Anthropic SDK)

```python
{
    "role": "tool",
    "content": [
        {
            "name": "function_name",
            "type": "text",
            "text": json.dumps(result_data)
        }
    ]
}
```

### Tool Result Format (OpenAI SDK)

```python
{
    "role": "tool",
    "tool_call_id": "call_abc123",  # Must match the tool_call.id from response
    "content": json.dumps(result_data)
}
```

**Critical**: The `tool_call_id` MUST match between the assistant's tool call and the tool result. Mismatched IDs cause MiniMax to reject the message with `"tool result's tool id() not found"`.

## 7. Parallel Tool Calling

M2.7 supports calling multiple tools in a single response. The model can issue multiple `<invoke>` blocks within a single `<minimax:tool_call>` wrapper.

### Performance Impact

Parallel tool calling yielded a **37% speed improvement** on SWE-Bench Verified evaluations by allowing file reads, searches, and other operations to execute simultaneously.

### Implementation

When the model returns multiple `tool_calls`, execute them all and return all results:

```python
if len(assistant_msg.tool_calls) > 1:
    # Execute all tool calls (optionally in parallel)
    results = []
    for tc in assistant_msg.tool_calls:
        result = execute_tool(tc.function.name, json.loads(tc.arguments))
        results.append({
            "role": "tool",
            "tool_call_id": tc.id,
            "content": json.dumps(result)
        })
    messages.extend(results)
```

### Tool Calling with Ollama

Ollama injects tool schemas into the system prompt and expects structured JSON responses. The model's native XML tool call format may need special handling. Ollama tool calling works by:

1. Providing tools as JSON in the `tools` field of the API request
2. The model returns `tool_calls` in the response message
3. Execute tools and return results with `role: "tool"`
4. Continue the loop until no tool calls remain

### Important Ollama Caveats

- Ollama's streaming implementation may not correctly return `tool_calls` delta chunks
- Always include instructions in the prompt for the model to output JSON when using structured output
- Without JSON instructions, token sampling may produce garbage or excessive whitespace
- For Docker: use `172.17.0.1:11434` not `host.docker.internal` on Linux

## 12. Known Quirks and Failure Modes

### Critical: XML Tool Calls Stripped by Frameworks

**Bug**: Many frameworks (OpenCode, Goose, etc.) route MiniMax to an Anthropic-compatible API but then strip the `<minimax:tool_call>` XML instead of parsing it. Tools become completely non-functional.

**Fix**: Use `--tool-call-parser minimax_m2` with vLLM/SGLang, or implement XML parsing in your framework.

Source: [OpenClaw #41839](https://github.com/openclaw/openclaw/issues/41839)

### Critical: Malformed Responses Poison Session History

**Bug**: M2.7 occasionally returns truncated JSON in tool call arguments (e.g., `{"maxChars": , "url": "ht..."}`). If a framework inserts a synthetic error result, MiniMax rejects it because the tool_call_id doesn't match, permanently corrupting the session.

**Fix**: Validate tool call arguments before executing. If malformed, retry the generation rather than inserting synthetic results. Implement session history cleanup.

Source: [OpenClaw #19523](https://github.com/openclaw/openclaw/issues/19523)

### Critical: Missing Required Parameters

**Bug**: M2 sometimes omits required parameters from tool calls. Example: calling a `write` tool with only `path` but missing required `content`.

**Fix**: Validate all required parameters before execution. If missing, return a descriptive error as the tool result and let the model retry.

Source: [OpenClaw #14596](https://github.com/openclaw/openclaw/issues/14596)

### High Impact: Overthinking Simple Tasks

**Bug**: M2.7's thinking cannot be disabled. Simple tasks like "say hello" take 2.2 seconds (vs 0.76s for Qwen). Complex tasks can take 195 seconds where competitors take 0.6 seconds due to unnecessary verification loops.

**Workaround**: Accept the latency tax. Do NOT try to reduce `max_tokens` to limit thinking — this truncates the answer, not the thinking. No official reasoning budget control exists for local deployments.

Source: [GitHub #77 — Thinking hurts agent scenarios](https://github.com/MiniMax-AI/MiniMax-M2/issues/77)

### Medium: System Prompt Constraint Violations

**Bug**: The thinking process can override explicit system prompt constraints. Example: a 60-character response limit produced 303 characters because the model prioritized "completeness" over brevity.

**Fix**: Add post-processing validation. Check outputs against constraints and re-prompt if violated.

Source: [GitHub #77](https://github.com/MiniMax-AI/MiniMax-M2/issues/77)

### Medium: Looping in Proxied Environments

**Bug**: M2.7 via OpenRouter can get stuck in loops, repeatedly generating similar responses. Observed across Windows and Linux, across multiple IDEs.

**Fix**: Ensure recommended sampling parameters are actually being transmitted (not just echoed in config). Session reaching 130K tokens without problems was reported after proper configuration.

Source: [OpenCode #15092](https://github.com/anomalyco/opencode/issues/15092)

### Low: Raw XML Leaking

**Bug**: In some frameworks, the model outputs raw `<minimax:tool_call>` XML in its text response because the opening tag was malformed or the parser missed it.

**Fix**: Implement fallback XML detection in your response handler.

Source: [Goose #5751](https://github.com/block/goose/issues/5751)

### Low: Go Mutex Obsession

**Quirk**: M2 can get "oddly obsessed with inserting mutexes into Go code that doesn't need them." Not a tool-calling issue but worth knowing for code generation tasks.

Source: [Verdent: What Is MiniMax M2](https://www.verdent.ai/guides/what-is-minimax-m2-coding)

## 13. Comparison: M2.7 vs Claude for Tool Calling

| Metric | M2.7 | Claude Opus 4.6 | Winner |
|--------|------|-----------------|--------|

## 14. Agent Loop Architecture

### Three-Phase Pattern (Proven with M2.5/M2.7)

```
1. PLAN — Describe changes file-by-file with dependency flags
2. DIFF — Output unified diffs in fenced code blocks
3. VERIFY — List code review checkpoints
```

This "PLAN-before-DIFF" behavior is M2.7's distinguishing feature — it surfaces cross-file conflicts early.

### Step Budget (Hard Ceiling)

```python
MAX_TOOL_STEPS = 15  # Hard ceiling on tool calls per session

step_count = 0
while step_count < MAX_TOOL_STEPS:
    response = call_model(messages, tools)
    if not response.tool_calls:
        break
    step_count += len(response.tool_calls)
    # ... execute tools ...

if step_count >= MAX_TOOL_STEPS:
    messages.append({"role": "user", "content": "BUDGET EXCEEDED. Summarize findings and stop."})
```

### Output Validation Layer

```python
for section in ["PLAN", "DIFF", "VERIFY"]:
    if section not in agent_output:
        raise ValueError(f"Missing '{section}' section. Re-prompt with phase labels.")

# Extract diffs from fenced blocks only
if "```diff" not in agent_output:
    raise ValueError("DIFF section contains no fenced diff blocks.")
```

### Context Management for Large Repos

- Rank files by keyword overlap with task goal
- Build context shards staying under 40K token budget
- Run two passes: hotspot shard first, secondary shard for dependency checks
- Process highest-relevance files first

### Guard Rails

- **Branch protection**: Reject writes to main/master/develop/staging/release at handler level
- **Tool whitelist**: Only expose tools the agent actually needs
- **Format drift detection**: Regex-validate output structure between response and execution
- **Loop detection**: The model itself discovered this optimization during self-evolution

## 15. Structured Output / JSON Mode

### Via Ollama (GBNF Grammar Enforcement)

Ollama uses llama.cpp GBNF grammars to constrain output to valid JSON. Pass a JSON schema via the `format` parameter:

```python
response = chat(
    model='minimax-m2.7:cloud',
    messages=[{"role": "user", "content": "Analyze this code"}],
    format={
        "type": "object",
        "properties": {
            "issues": {"type": "array", "items": {"type": "string"}},
            "severity": {"type": "string", "enum": ["low", "medium", "high"]}
        },
        "required": ["issues", "severity"]
    }
)
```

**Important**: Always include instructions in the prompt to output JSON. Without them, the grammar constraint forces valid JSON but the content may be poor (lots of whitespace, wrong fields).

### Via OpenRouter

OpenRouter's Response Healing fixes JSON syntax errors (trailing commas, unclosed braces) but does NOT fix schema adherence issues (wrong field names, missing required properties, wrong types).

### Via vLLM/SGLang

Tool calls are automatically structured through the `--tool-call-parser minimax_m2` flag. For non-tool structured output, use the model's native capabilities with explicit format instructions in the prompt.

## 16. Prompting Strategies for Non-Claude Models

### General Principles for Open-Source Models

1. **Be more explicit than you think necessary**. Open models cannot infer as much from context as Claude.

2. **Use the DON'T section aggressively**. Prevent creative drift by explicitly listing what NOT to do.

3. **Show, don't tell**. One or two examples beat paragraphs of instructions. For complex formats, provide a complete template.

4. **Keep tool schemas simple**. Descriptions should be concise. Use obvious parameter names. Include `enum` for constrained choices.

5. **Validate everything**. Post-process all outputs. Check required fields, types, and constraints. Re-prompt on failure.

### Making Tool Schemas Clearer

```python
# BAD: Vague description
{"name": "search", "description": "Search for things"}

# GOOD: Specific, with format guidance
{
    "name": "search_codebase",
    "description": "Search the repository for code matching a pattern. Returns file paths and matching lines. Use grep-style regex patterns.",
    "parameters": {
        "type": "object",
        "properties": {
            "pattern": {
                "type": "string",
                "description": "Regex pattern to search for, e.g. 'def process_.*\\(' or 'import json'"
            },
            "file_glob": {
                "type": "string",
                "description": "File pattern to limit search, e.g. '*.py' or 'src/**/*.ts'. Default: all files."
            }
        },
        "required": ["pattern"]
    }
}
```

### Few-Shot Examples in System Prompts

For tool-calling agents, include an example exchange:

```
Example interaction:
User: "What's the weather in Tokyo?"
Assistant thinking: I need to call get_weather with location="Tokyo, Japan"
[Tool call: get_weather(location="Tokyo, Japan", unit="celsius")]
Tool result: {"temperature": 22, "condition": "Partly cloudy"}
Assistant: The weather in Tokyo is 22C and partly cloudy.
```

### Temperature Considerations

- M2.7: Use temperature=1.0 (trained at this temperature)
- Other open models: Temperature 0.1-0.3 for structured/tool-calling tasks
- Ollama recommendation: Temperature 0.1 for consistency in tool-calling scenarios

## 18. Cortex-Specific Recommendations

Based on all research, here are the actionable recommendations for Cortex overnight missions:

### 1. Always Preserve Think Tags

In Cortex's agent loop, when building conversation history, NEVER strip `<think>` tags or `reasoning_details`. This single issue causes the most degradation.

### 2. Use the Right Sampling Parameters

```python
CORTEX_M27_PARAMS = {
    "temperature": 1.0,    # Do NOT lower this
    "top_p": 0.95,
    "top_k": 40,
    "max_tokens": 8192,    # Generous to avoid truncation
}
```

### 3. Implement Tool Call Validation

Before executing any tool call from M2.7:
- Check all required parameters are present
- Validate JSON parse-ability of arguments
- If malformed, return error to model (don't retry silently, don't insert synthetic results)

### 4. Set a Step Budget

`MAX_TOOL_STEPS = 15` per mission. When exceeded, force summarization.

### 5. Post-Process All Outputs

Regex-validate expected output structure. Re-prompt with specific missing section names if validation fails.

### 6. Use Anthropic-Compatible API When Possible

The Anthropic API natively supports interleaved thinking. The OpenAI API requires workarounds (`reasoning_split`) and many frameworks silently drop the reasoning. If using LiteLLM:

```yaml
# Prefer this:
api_base: https://api.minimax.io/anthropic/v1/messages

# Over this:
api_base: https://api.minimax.io/v1
```

### 7. Prompt Structure for Overnight Missions

```
You are Cortex, an autonomous agent conductor. You have access to the following tools.

RULES:
1. Think carefully before each tool call.
2. You may call multiple tools in a single turn.
3. After each tool result, reflect on whether you need more information.
4. When you have enough information, provide your final answer.
5. Do NOT repeat tool calls with identical parameters.
6. If a tool call fails, try a different approach rather than retrying the same call.

TASK: [mission description]

CONSTRAINTS: [specific limits, file paths, branch rules]

DON'T: [explicit exclusions]
```

### 8. Handle Session Corruption

If M2.7 returns a malformed response:
1. Do NOT insert synthetic tool results
2. Trim the last malformed message from history
3. Retry the generation
4. If corruption persists after 3 retries, log and escalate

### 9. Accept the Latency Tax

M2.7 will think before every tool call. This is by design. The thinking improves quality but adds 1-3 seconds per turn. For overnight missions where time is abundant, this is an acceptable trade-off for the 17-21x cost savings over Claude.

### 10. Self-Evolution Pattern

Cortex should track failure trajectories and evolve its own scaffold over time:
- Log every failed tool call (parameters, error, context)
- After N failures of the same type, generate a workflow rule to prevent it
- Periodically audit workflow rules for effectiveness
- This is the exact pattern M2.7 used to improve itself by 30%
